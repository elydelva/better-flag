## When to use

Use rollout when:
- **Gradual release** — Enable a feature for 10%, then 50%, then 100% of users
- **A/B testing** — Split traffic between control and treatment
- **Canary** — Roll out to a percentage before full release

**Deterministic**: Same user + same flag = same result every time. No random flickering.

## How it works

1. A hash is computed from `flagKey` + `seed` (or `context.userId`).
2. The hash maps to a percentage bucket (0–100).
3. If the bucket is within `rollout.percentage`, the flag is **on** (or variant A); otherwise **off** (or variant B).

## Rollout config

```ts
interface RolloutConfig {
  percentage: number;  // 0–100
  seed?: string;       // e.g. "userId" for consistent bucketing
}
```

- **percentage**: 0–100. Percentage of users who get the "on" or primary variant.
- **seed**: Attribute to use for hashing. Default: `"userId"`. Use `"userId"` for per-user consistency.

## Example

```ts
{
  key: "new-feature",
  type: "boolean",
  defaultValue: false,
  rollout: {
    percentage: 25,
    seed: "userId",  // use context.userId
  },
}
```

25% of users (by consistent hash) get `true`; 75% get `false`.

## hashToPercentage

For custom logic, use the exported helper:

```ts
import { hashToPercentage } from "@better-flag/core";

const pct = hashToPercentage("user-123", "new-feature", "salt");
// 0–100
```

## Variant rollout

For `type: "variant"`, weights determine the split. Same hashing applies: each user gets a stable variant. See [Flag](/docs/flag#variant-flags).
